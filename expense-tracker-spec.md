# Трекер трат — спецификация проекта

Сводка всех решений, принятых в обсуждении. Использовать как техзадание при
дальнейшей доработке приложения.

## Концепция

Личное веб-приложение для учёта расходов на Android (PWA), без сервера и без
регистрации. Все данные хранятся локально в браузере (IndexedDB), синхронизация
между устройствами не требуется — только один пользователь, одно устройство,
с ручным экспортом/импортом JSON как способом бэкапа и переноса.

Причина отказа от бэкенда: не хочется поддерживать сервер ради личного
инструмента. Если в будущем понадобится multi-device или multi-user —
можно добавить синхронизацию (например, Supabase) поверх текущей архитектуры
как отдельный слой, не переписывая фронтенд с нуля (паттерн "local-first").

Автоматический парсинг чеков по фото (OCR) сознательно отложен — сначала
нужно обкатать ручной/полуавтоматический ввод.

## Технический стек

- **React** (без роутинга — переключение экранов через `useState`, т.к. экранов
  немного и глубокая навигация с URL не нужна)
- **Dexie.js** — обёртка над IndexedDB
- **@dnd-kit** (core + sortable) — drag-and-drop сортировка категорий/подкатегорий
  (мышь, тач, клавиатура)
- **Vite** — сборка
- **vite-plugin-pwa** — установка на Android как приложение, офлайн-работа,
  `registerType: 'autoUpdate'` (service worker сразу активирует новую версию
  в фоне через `skipWaiting`+`clientsClaim` — см. грабли ниже)
- **Хостинг — развёрнут и используется**: репозиторий на GitHub
  (`UCaTiK/expense-tracker`), подключён к Vercel (бесплатный тариф) с
  автодеплоем при каждом пуше в `main`. Постоянный домен проекта не меняется
  между деплоями (в отличие от отдельных preview-ссылок на конкретный коммит).

Стек выбран с учётом того, что весь код пишется через AI ("вайб-кодинг"),
без опыта фронтенда — React как наиболее предсказуемый для AI-генерации
вариант.

## Модель данных

```
Purchase (покупка / поход в магазин):
- id
- date (timestamp; время суток внутри дня не считается значимым — см. ниже)
- place (место, с автодополнением по истории)
- totalPaid (реально уплаченная сумма)
- categoryId (категория верхнего уровня — используется в быстром режиме
  или как fallback; при детальном режиме реальная разбивка берётся из items)
- note (опционально)
- tagIds: [id] (теги, к покупке можно прикрепить несколько — выбираются
  только из уже существующих, создание нового тега возможно только в
  Настройки → Управление → Теги, не прямо из формы покупки)
- needsDetail (true, если нет ни одной позиции — покупка "не детализирована")

PurchaseItem (позиция внутри покупки, только в детальном режиме):
- id, purchaseId
- name (название товара, необязательно для заполнения)
- subcategoryId (подкатегория — лист дерева категорий)
- amount

Category (дерево: категория верхнего уровня + подкатегории):
- id, name, icon, color
- parentId (null = категория верхнего уровня)
- isDefault (предустановленная или пользовательская)
- isArchived (архивация вместо жёсткого удаления — не теряет историю по
  старым покупкам; архивация категории верхнего уровня каскадно архивирует
  её подкатегории; архивные категории скрыты из пикеров при выборе новой
  покупки, но по-прежнему корректно отображаются в уже сохранённых записях)
- sortOrder (ручная сортировка drag-n-drop реализована — отдельно для
  категорий верхнего уровня и отдельно внутри каждого родителя для
  подкатегорий)
- defaultKey (только у предустановленных категорий — стабильный внутренний
  ключ для дедупа и миграции набора дефолтов; не показывается в UI и
  переживает переименование пользователем)

Tag (справочник тегов):
- id, name, color
(в покупке — просто массив tagIds, без отдельной junction-таблицы:
 IndexedDB/Dexie хорошо работает с массивами напрямую)
По умолчанию засеяны 3 тега: "Ежемесячно", "Еженедельно", "Ежегодно".

CategoryHint (словарь автоподсказки подкатегории по названию позиции):
- normalizedName, subcategoryId, lastUsed
Таблица в схеме присутствует, но фича сознательно отключена: подкатегория
всегда выставляется вручную, автоматически может подставляться только
верхнеуровневая категория (см. экран "Добавление/редактирование покупки").
Задел на будущее — полноценный справочник "позиция → категория/подкатегория"
будет отдельной фичей позже.

Place (справочник мест для автодополнения):
- name, lastUsed, defaultCategoryId (категория, которая подставляется
  автоматически при быстром добавлении покупки в этом месте — запоминается
  только по быстрым покупкам, детальные её не перезаписывают)
Место можно переименовать, удалить или поменять его связку с категорией в
Настройки → Управление → Места — это влияет только на автодополнение,
существующие покупки не меняются.

Account (задел на будущее, ещё не реализовано):
- id, name, initialBalance
Purchase получит необязательное поле accountId, когда дойдёт очередь.
Поле не блокирует остальную модель — можно добавить как миграцию позже.
```

Ключевой архитектурный принцип: категория/тег/место — всегда связь по `id`,
никогда не текстовое поле напрямую. Это даёт бесплатное переименование,
защиту от опечаток и корректную агрегацию в аналитике.

Второй ключевой принцип: категория "живёт" на уровне позиции (`PurchaseItem`),
а не всей покупки — так поход в Ашан (где продаются и продукты, и одежда)
не ломает категоризацию: просто несколько позиций с разными подкатегориями
в рамках одной покупки.

Сортировка покупок внутри одного дня — **по алфавиту (по месту)**, не по
времени: `purchase.date` не несёт надёжного времени суток (оно сбрасывается
на полночь при любом ручном выборе даты через календарь), а трекер трат не
следит за временем совершения покупки в течение дня в принципе. Между
разными днями сортировка обычная — от новых к старым.

## Набор категорий по умолчанию

8 категорий верхнего уровня, у каждой свой акцентный цвет и иконка:

1. **Продукты** — Готовая еда, Овощи / Фрукты, Мясо / Рыба / Яйца,
   Хлеб / Выпечка, Молочка / Сыр, Бакалея, Напитки, Алкоголь, Сладости,
   Дополнение, Прочее, Без категории
2. **Рестораны** — Кафе / фастфуд, Рестораны, Доставка еды, Напитки, Алкоголь
3. **Одежда** — Одежда, Обувь, Аксессуары
4. **Транспорт** — Такси, Общественный транспорт, Каршеринг
5. **Дом и быт** — ЖКХ, Связь и интернет, Ремонт, Товары для дома
6. **Здоровье** — Аптека, Врачи, Уход
7. **Развлечения** — Досуг, Развлечения, Игры, Хобби
8. **Прочее** — Подарки, Прочее

При смене этого набора в коде (правка `DEFAULT_CATEGORIES` в `db/seed.js`)
существующая база **не остаётся на старом наборе** — миграция
(`migrateDefaultCategoriesV2`, гоняется на каждом старте) переименовывает и
пересортировывает уже существующие дефолтные категории верхнего уровня на
месте (id не меняется, старые покупки не ломаются), архивирует подкатегории,
которых больше нет в новом наборе (не удаляет — иначе покупки, ссылавшиеся на
них, сломались бы), и добавляет новые. Чистая установка сразу сеется новым
набором через `seedDatabase()`, минуя миграцию.

## Экраны

Навигация: нижняя панель ровно с 3 вкладками — **Аналитика, Главная,
Настройки** (Главная посередине). Категории, Теги и Места — не вкладки, а
отдельные страницы внутри Настроек, каждая со своей кнопкой "назад".

1. **Главная** — сумма за месяц сверху (карточка с навигацией по месяцам
   стрелками/свайпом, движение ограничено реальным диапазоном дат в базе —
   см. ниже), список покупок сгруппирован по дням (новые сверху, дневной
   подытог в заголовке группы), внутри дня — по алфавиту места. В строке
   покупки: иконка категории, место, категория/подкатегория (или пометка "не
   детализировано"), сумма, теги (если есть, максимум 1-2 видно, дальше "+N").
   Кнопка "Добавить покупку" — полноширинная скруглённая, зафиксирована над
   нижней навигацией (не поверх неё, не перекрывает список), видна только на
   Главной.
2. **Добавление/редактирование покупки** — одна форма для создания и правки
   (единственная разница — новый id или обновление существующего).
   Переключатель "Быстро / Детально":
   - *Быстро*: место, дата, категория (сетка иконок верхнего уровня), сумма.
   - *Детально*: место, дата, список позиций (название необязательно,
     категория/подкатегория через кастомный пикер, сумма). Новая позиция не
     подставляет сумму автоматически — вместо этого под пустым полем
     показывается тап-подсказка "остаток от указанной суммы" (для первой
     позиции это вся сумма целиком), по тапу подставляется в поле. Категория
     новой позиции по умолчанию наследуется от предыдущих позиций этой же
     покупки (самая частая), подкатегория — всегда только вручную. Сравнение
     суммы позиций с реально уплаченной суммой — при расхождении показывается
     предупреждение с точной разницей и кнопкой "Указать как уплачено"
     (подставляет в поле общей суммы сумму всех позиций); расхождение не
     блокирует сохранение. Теги выбираются только из уже существующих
     (создание нового тега — не отсюда, см. Настройки → Управление → Теги).
3. **Просмотр покупки** — место, дата, крупная сумма, теги, разбивка по
   категориям (полоски, **сортировка по убыванию суммы**), список позиций с
   цветовой привязкой к категориям. Кнопки: единая "Редактировать" (независимо
   от того, детализирована покупка или нет) и удаление с подтверждением.
4. **Аналитика** — переключатель периода (неделя/месяц/год, скруглённый
   сегментированный контрол), навигация между периодами свайпом влево/вправо
   + стрелки. **Неделя — с понедельника, месяц — с 1 календарного числа.**
   Навигация стрелками ограничена реальным диапазоном дат в базе: вперёд/
   назад можно двигаться только туда, где дальше действительно есть хоть одна
   покупка — при этом пустой период *между* двумя периодами с данными (напр.
   пустой февраль между январём и июлем) всё равно доступен для просмотра,
   просто показывает пустое состояние; стрелка скрывается только на
   истинной границе диапазона. Итог с сравнением к предыдущему периоду в
   процентах. Список категорий по убыванию суммы, у каждой — своя полоска и
   свой % изменения к предыдущему периоду. Счётчик недетализированных
   покупок за период. Для **текущего незавершённого периода** — переключатель
   "с начала периода" (честное сравнение на одинаковом отрезке дат) / "весь
   период" (грубее, но даёт масштаб). Для уже прошедших периодов сравнение
   всегда полное, без переключателя. Если данных за предыдущий период нет
   вообще — сообщение "недостаточно данных для сравнения" вместо
   некорректного/бессмысленного процента.

   **Тап по категории** открывает её отдельную страницу: тот же переключатель
   периода и навигация стрелками сверху (границы навигации здесь считаются по
   данным именно этой категории, а не по всем покупкам), сумма за период по
   категории, сворачиваемая/разворачиваемая разбивка по подкатегориям
   (позиции без выбранной подкатегории и недетализированные покупки этой
   категории попадают в общий бакет "Без подкатегории"), и список покупок этой
   категории за период — сгруппирован по дням и отсортирован по алфавиту
   внутри дня, как на Главной.
5. **Категории** (внутри Настроек → Управление) — каждая категория верхнего
   уровня — отдельная скруглённая карточка (не общий плоский список), её
   подкатегории при раскрытии остаются внутри той же карточки. И у категорий
   верхнего уровня, и у подкатегорий: drag-хендл для сортировки (перетаскивание
   мышью/тачем/клавиатурой через @dnd-kit, в своих пределах — подкатегории
   сортируются только внутри родителя) и меню "..." (переименовать, сменить
   иконку, сменить цвет, архивировать/восстановить). Архивные — приглушены, с
   пометкой "В архиве". Отступ иконки категории одинаков со всех трёх сторон
   (слева/сверху/снизу).
6. **Теги** (внутри Настроек → Управление) — список тегов, тап по тегу
   открывает редактирование (название, цвет), справа — иконка корзины для
   быстрого удаления с подтверждением; удаление тега убирает его из всех
   покупок, где он был проставлен.
7. **Места** (внутри Настроек → Управление) — список мест с полем поиска,
   тап по месту открывает редактирование (переименование, смена связанной
   категории), справа — иконка корзины для быстрого удаления с
   подтверждением. Любое изменение здесь влияет только на автодополнение и
   подсказку категории при новой покупке — уже сохранённые покупки не
   меняются.
8. **Настройки** — блоки: "Данные" (экспорт/импорт JSON, с датой последнего
   бэкапа), "Управление" (Категории, Теги, Места, Счета — с пометкой "скоро"),
   "Хранилище" (число сохранённых покупок и точный размер данных в Б/КБ/МБ,
   посчитанный напрямую из содержимого таблиц IndexedDB, а не через
   `navigator.storage.estimate()` — та величина включает в себя весь кэш
   service worker'а и вводит в заблуждение), отдельно — деструктивное
   "Очистить все данные" с подтверждением (после очистки категории по
   умолчанию создаются заново).

## Дизайн

Тёмная тема (единственная — светлая не нужна), простой современный флэт-дизайн:
плоские поверхности, минимум границ (0.5px, полупрозрачные), без теней и
градиентов, скруглённые карточки (10-16px радиус).

Нижняя навигация — плавающая скруглённая карточка (тот же скруглённый
сегментированный стиль, что и переключатель периода в Аналитике: активная
вкладка получает заливку-пилюлю), инсет от краёв экрана на те же 16px, что и
остальной контент страниц. Кнопка "Добавить покупку" — отдельная такая же
скруглённая карточка над навигацией (не круглая плавающая кнопка поверх неё,
как было раньше), полной ширины, видна только на Главной.

Цвета — нейтральные серые для фона/поверхностей/текста (без цветового
оттенка — важно, чтобы фон не отдавал зеленью или коричневым). Акцентные
цвета — приглушённая палитра (зелёный/оранжевый/фиолетовый/синий/красный/
розовый/бирюзовый/коралловый), по одному цвету на категорию, настраивается
пользователем.

## Известные технические грабли (уже наступили, зафиксировано решение)

- `crypto.randomUUID()` требует secure context (https/localhost) — не работает
  при открытии по обычному http с телефона в локальной Wi-Fi сети. Генерация
  id должна идти через `crypto.getRandomValues` вручную, не через `randomUUID`.
- `React.StrictMode` в dev-режиме двойно вызывает эффекты — при заполнении
  БД дефолтными данными на первом запуске это привело к задвоению категорий.
  Решение: не использовать StrictMode + защита от повторного запуска сидинга
  через singleton-промис + функция дедупликации, которая подчищает уже
  возникшие дубли при каждом старте.
- IndexedDB не индексирует записи, где индексируемое поле равно `null`
  (это ограничение спецификации IndexedDB) — запросы вида
  `where('parentId').equals(null)` не сработают для категорий верхнего
  уровня; нужно получать их через `toArray()` + `filter()`, а не через индекс.
- Service worker (`registerType: 'autoUpdate'`, `skipWaiting`+`clientsClaim`)
  подхватывает новую версию тихо и мгновенно в фоне при следующем открытии
  приложения, но уже открытая вкладка/установленное PWA не перезагружается
  сама — если деплой происходит прямо во время активной работы с формой, это
  может оборвать сессию и потерять несданные изменения. Смягчение: при
  активной разработке лучше закрывать и заново открывать приложение, зная,
  что только что был пуш; полностью убрать риск можно переходом на
  `registerType: 'prompt'` с явным баннером "доступно обновление" — осознанно
  не сделано, обычный риск сочтён приемлемым.
- `purchase.date` не несёт надёжного времени суток — см. раздел про
  сортировку внутри дня.
