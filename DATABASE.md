# База данных — модели

Технический справочник по схеме IndexedDB (через Dexie.js). Для общего
описания фич и экранов — см. `expense-tracker-spec.md`.

## Обзор

- Одна база: `expenseTrackerDB`, `db.version(1)` (миграций схемы пока не было).
- Определение — `src/db/db.js`. Каждая таблица имеет свой модуль-репозиторий
  в `src/db/*.js` с функциями чтения/записи; экраны читают через
  `useLiveQuery`-хуки в `src/hooks/*.js`, которые реактивно перерисовываются
  при любом изменении данных.
- Все первичные ключи, кроме `tags`/`places` — строки, генерируются вручную
  через `crypto.getRandomValues` (`src/lib/uuid.js`), **не** через
  `crypto.randomUUID()` — тот требует secure context и не работает при
  открытии по обычному `http` с телефона в локальной Wi-Fi сети.
- Связи между таблицами — всегда по `id`, никогда по названию/тексту
  напрямую (даёт бесплатное переименование и защиту от опечаток).

```
db.version(1).stores({
  purchases:     'id, date, categoryId',
  purchaseItems: 'id, purchaseId, subcategoryId',
  categories:    'id, parentId, sortOrder',
  tags:          'id, &name',
  categoryHints: 'normalizedName, subcategoryId, lastUsed',
  places:        'name, lastUsed',
});
```

## purchases

Одна запись = один поход в магазин / одна покупка.

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (PK) | |
| `date` | number (timestamp, мс) | Время суток внутри дня не надёжно: сбрасывается на полночь при выборе даты через календарь (см. `src/lib/format.js: fromDateInputValue`). Сортировка внутри одного дня поэтому идёт по алфавиту места, а не по этому полю. |
| `place` | string | Свободный текст с автодополнением из `places` |
| `totalPaid` | number | Реально уплаченная сумма |
| `categoryId` | string \| null | FK → `categories.id`. Источник истины в быстром режиме; в детальном — fallback для строки списка (верхнеуровневая категория позиции с наибольшей суммой) |
| `note` | string | Опционально |
| `tagIds` | string[] | FK[] → `tags.id`. Обычный массив, без `multiEntry`-индекса — ни один экран не фильтрует покупки по тегу, только отображает |
| `needsDetail` | boolean | `true`, если нет ни одной позиции ("не детализировано") |
| `accountId` | null | Зарезервировано под будущую таблицу `Account`, пока всегда `null` |

Индекс: `id, date, categoryId`.

## purchaseItems

Позиция внутри покупки — существует только для детального режима.

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (PK) | |
| `purchaseId` | string | FK → `purchases.id` |
| `name` | string | Название товара, необязательно |
| `subcategoryId` | string | FK → `categories.id`. Может указывать и на настоящую подкатегорию (`parentId !== null`), и на категорию верхнего уровня напрямую, если пользователь выбрал только категорию без подкатегории |
| `amount` | number | |

Индекс: `id, purchaseId, subcategoryId`. Аналитика и просмотр покупки
запрашивают позиции по `purchaseId` (через `.anyOf([...])` при пакетной
загрузке за период) и резолвят `subcategoryId` через `resolveTopCategoryId`
(`src/lib/categoryTree.js`) для группировки по верхнеуровневой категории.

## categories

Дерево: категория верхнего уровня + её подкатегории, одна и та же таблица
для обоих уровней.

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (PK) | |
| `name` | string | |
| `icon` | string \| null | Ключ в `ICON_REGISTRY` (`src/lib/icons.js`); всегда `null` у подкатегорий |
| `color` | string \| null | Один из `CATEGORY_COLORS` (`src/lib/colors.js`); всегда `null` у подкатегорий |
| `parentId` | string \| null | `null` = категория верхнего уровня. IndexedDB не индексирует записи с `null` в индексируемом поле — верхнеуровневые категории всегда получают через `toArray()+filter()`, не через `.where('parentId')` |
| `isDefault` | boolean | Предустановленная (сидинг) или созданная пользователем |
| `isArchived` | boolean | Архивация вместо удаления — не ломает историю уже сохранённых покупок/позиций, которые на неё ссылаются. Архивация категории верхнего уровня каскадно архивирует её текущие активные подкатегории |
| `sortOrder` | number | Ручная сортировка (drag-and-drop) — отдельно для категорий верхнего уровня и отдельно внутри каждого родителя для подкатегорий |
| `defaultKey` | string (только у дефолтных) | Стабильный внутренний ключ для дедупа и миграции набора дефолтов между версиями кода; не показывается в UI, переживает переименование |

Индекс: `id, parentId, sortOrder`.

## tags

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (PK) | |
| `name` | string (уникально) | |
| `color` | string | Один из `CATEGORY_COLORS` |

Индекс: `id, &name` (`&` — уникальный индекс). По умолчанию засеяны 3 тега:
"Ежемесячно", "Еженедельно", "Ежегодно". Удаление тега (`deleteTag`) чистит
его из `tagIds` всех покупок — `tagIds` не индексируется, поэтому покупки
сканируются в памяти (`toArray()+filter()`).

## categoryHints (в схеме есть, функция не подключена)

| Поле | Тип | Описание |
|---|---|---|
| `normalizedName` | string (PK) | Название позиции в нижнем регистре, без цифр/процентов |
| `subcategoryId` | string | FK → `categories.id` |
| `lastUsed` | number | timestamp |

Задумывался как самообучающийся словарь "название позиции → подкатегория",
но фича сознательно отключена: подкатегория позиции всегда выставляется
вручную, автоматически может подставляться только верхнеуровневая категория
(по самой частой категории среди уже добавленных позиций той же покупки).
Таблица и индекс остаются в схеме как задел на будущее; ничего в неё сейчас
не пишется.

## places

| Поле | Тип | Описание |
|---|---|---|
| `name` | string (PK) | Сам текст места — таблица используется только для автодополнения, покупки хранят место как обычную строку, не FK |
| `lastUsed` | number | timestamp последнего использования — для сортировки автодополнения |
| `defaultCategoryId` | string \| null | FK → `categories.id`. Подставляется в форму при быстром добавлении покупки в этом месте; пишется только быстрыми покупками — детальные не перезаписывают уже выученное значение |

Индекс: `name, lastUsed`. Переименование/удаление места или смена его связи
с категорией (Настройки → Управление → Места) влияет только на
автодополнение и подсказку — уже сохранённые покупки не меняются, так как
`purchase.place` — это независимая строка, а не ссылка на `places`.

## Экспорт / импорт (`src/db/exportImport.js`)

Бэкап — один JSON-файл со всеми 6 таблицами:

```json
{
  "schemaVersion": 1,
  "exportedAt": "2026-07-13T12:00:00.000Z",
  "data": {
    "purchases": [...],
    "purchaseItems": [...],
    "categories": [...],
    "tags": [...],
    "categoryHints": [...],
    "places": [...]
  }
}
```

- `exportData()` — читает все таблицы, скачивает JSON, пишет
  `localStorage['expense-tracker:lastBackupAt']` (не в Dexie — это
  UI-метаданные, не финансовые данные).
- `importData(json)` — validate `schemaVersion` + наличие всех 6 ключей,
  затем в одной транзакции: `clear()` каждой таблицы + `bulkPut()` из файла
  (полная замена, не слияние).
- `clearAllData()` — очищает все 6 таблиц и заново вызывает
  `seedDatabase()`, иначе приложение осталось бы без категорий/тегов.

## Сидинг и миграция дефолтных категорий (`src/db/seed.js`)

- `seedDatabase()` — сеет `DEFAULT_CATEGORIES` (8 категорий верхнего уровня
  с подкатегориями) и `DEFAULT_TAGS`, только если соответствующая таблица
  пуста. Гоняется один раз через `ensureSeeded()` (singleton-промис — защита
  от двойного вызова).
- `dedupeDefaultCategories()` — на каждом старте подчищает дубли дефолтных
  категорий (по `defaultKey`), если они всё же возникли, и перевешивает
  ссылки `purchaseItems.subcategoryId` / `purchases.categoryId` с дублей на
  оставшийся экземпляр.
- `migrateDefaultCategoriesV2()` — на каждом старте приводит уже
  существующую базу в соответствие с текущим `DEFAULT_CATEGORIES`: переименовывает
  и пересортировывает категории верхнего уровня на месте (id не меняется),
  архивирует подкатегории, которых больше нет в наборе (не удаляет —
  иначе `purchaseItems`, ссылающиеся на них, сломались бы), добавляет новые.
- `repairDefaultCategoryStyling()` — на каждом старте чинит `icon`/`color`
  дефолтных категорий, если они пустые или не входят в известный список
  (страховка от частично записанных данных со старой вкладки).

Порядок вызова при загрузке приложения (`src/main.jsx`): `ensureSeeded` →
`dedupeDefaultCategories` → `migrateDefaultCategoriesV2` →
`repairDefaultCategoryStyling`.
